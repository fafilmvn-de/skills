1What this is

A single command that turns "I need a new campaign" into a production-ready 6-notebook Databricks pipeline.

Campaign Orchestrator is a Copilot CLI skill bundle living under skills/. It drives the canonical campaign pipeline — six notebooks (data-prep → PO aggregation → hand-off → leads-list v1 → leads-list v2 → CE inputs) — without forcing you to rewrite boilerplate. The shared boilerplate already lives in generic-campaigns/template/ and generic-campaigns/lib/; the orchestrator's job is to interview you for what's campaign-specific, verify your data assumptions against the live workspace, and gate every change behind a real run.

Why bother?

Past campaigns took 3–6 days to bootstrap. With the orchestrator, scaffolding + intake + first stage typically take under an hour. The schema-verifier catches hallucinated column names before they ever reach a cluster.

2Mental model

Hold these three ideas in mind:

1 · The framework owns the boilerplate

generic-campaigns/template/ ships the 6 notebooks with the common eligibility CTEs (in-force status, age band, plan-code filter, corporate exclusion, channel filter). You almost never touch the common code.

2 · The intake is the manifest

One human-readable file — docs/intake.md — captures everything campaign-specific in 8 sections (metadata, eligibility, exclusions, segmentation, hand-offs, outputs). No YAML, no JSON config. If it's not in the intake, the orchestrator won't write code for it.

3 · The orchestrator coordinates skills

It is not a single LLM brain doing everything. It dispatches to four specialised sub-skills, each owning one verb of the lifecycle.

4The cast — four skills

Each skill has its own SKILL.md under skills/<name>/. The orchestrator routes work to the right one.

campaign-orchestrator entry point

Parses /campaign <verb>, manages the campaign folder lifecycle, runs the intake interview, and dispatches to the others.

skills/campaign-orchestrator/

intake-extractor

Reverse-engineers a draft intake.md from an existing campaign — invaluable when cloning a similar batch. Flags exclusions worth promoting into lib/.

skills/intake-extractor/

schema-verifier anti-hallucination

Runs DESCRIBE TABLE against the workspace via the SQL warehouse. Caches into lt-memory/schemas/ for 7 days. Refuses to confirm columns that don't exist.

skills/schema-verifier/

deploy-verifier

Closes the loop: git push → wait for workspace Git folder to pull → submit one-shot job → assert row counts. Never uses databricks workspace import.

skills/deploy-verifier/

6Happy-path walkthrough

A new batch for an existing campaign, cloned from a prior batch.

Step 0 · Extract the reference intake (optional but recommended)

If batch-2 doesn't already have a clean intake.md (some older campaigns don't), generate a draft:

/campaign extract-intake agency-campaigns//

The skill reads the 4 notebooks + any existing intake + audit-trail, and emits a draft with <!-- inferred from: ... --> provenance comments. It will also flag any exclusions present in the reference that aren't in lib/exclusions.py — for instance, excl_shared_phone, which appears in batch-1 but not batch-2.

Step 1 · Initialise the new batch

/campaign init   --reference agency-campaigns//

This copies generic-campaigns/template/ to agency-campaigns///, seeds docs/intake.md from the reference, and walks you through any remaining <!-- TODO --> placeholders one question at a time. Confirm each answer; the orchestrator edits the intake in place.

What about new campaigns with no reference?

Drop the --reference flag. The init verb will ask every question in the intake interview from scratch.

Step 2 · Author stage 01 (data-prep)

/campaign stage data-prep

The orchestrator loads skills/campaign-orchestrator/prompts/stage_data_prep.md, reads sections 2 + 3 of the intake, runs every referenced table through schema-verifier, and then inserts the campaign-specific eligibility CTEs / WHERE conditions into 01_*_UC.py — all inside a sentinel block. The common eligibility code is untouched.

Step 3 · Verify stage 01 on the cluster

/campaign verify data-prep

Behind the scenes this calls deploy-verifier:

  • Confirms your working tree is clean.
  • git push to GitHub.
  • Polls the Databricks Git folder (Repos id 985143326884916) until its HEAD SHA matches your local HEAD.
  • Submits a one-shot job against .
  • Polls run state; on success, runs SELECT COUNT(*) FROM <output> via the SQL warehouse and asserts ≥ your minimum.
  • Appends the result to docs/audit-trail.md.

Step 4 · Repeat for stages 02 → 06

Same pattern: /campaign stage <name>, then /campaign verify <name>. Each verify pass is your fail-loud gate before the next stage. If something is wrong (row count too low, schema drift, etc.), fix it locally, commit, and re-run verify.

Step 5 · Generate docs

/campaign stage docs

This is the only stage that runs after the pipeline. It reads the audit trail and funnel artefacts to render docs/README.md, docs/DOC.md, and a handovers/<CAMPAIGN_CODE>.html page.

7Stage-by-stage authoring

NB01

data-prep

policy-grain

Filters in-force policies by plan family / nurturing range / cash-value / tenure rules, joins PO details + city. Outputs <campaign>_<batch>_pol_final.

  • Common eligibility is framework-owned (status, age, plan, corporate).
  • Campaign-specific additions live inside /* campaign-specific:data-prep:START */ markers — typically extra WHERE clauses or a new CTE.
  • Schema-verifier pre-checks every column you reference.
NB02

po-aggregation

PO-grain

Collapses policy-grain to one row per PO. Applies servicing-agent tier rules, computes per-PO aggregates (sum APE, max tenure, total cash), and assigns the segmentation tier.

  • Custom tier acceptance list, retention-override flag, segmentation CASE expression — all sentinel-wrapped.
  • Channel enum is closed {Agency, VTB}; closed-bank policies serviced by Agency staff are mapped to Agency (per ADR 0002).
NB03

handoff-mapping

may be skipped

If your intake's section 7 says "no external hand-off", this stage is a near-no-op pass-through. Otherwise, it joins the hand-off CSV (uploaded to a Volume) and applies KEEP/DROP/DEFER decisions per PO.

NB04 + NB05

leads-list

shared notebook

The core leads-list builder. Adds enrichment columns, applies framework + campaign exclusions, builds the funnel, and calls assert_leads_list_schema(df) — fail-loud if the output schema diverges from the contract.

  • NB04 = v1 (pre-launch); NB05 = v2 (near-launch refresh). NB05 is a 2-cell %run shim with RUN_TAG=v2. Never duplicate logic into NB05.
  • If a campaign-specific exclusion appears in ≥2 campaigns, the orchestrator will offer to promote it into lib/exclusions.py.
NB06

engine-inputs

body is sacred

A %run shim around /agency-campaigns/_campaign_input_files.py. Produces the four CE input CSVs (ENGINE_MEMBER / ENGINE_PROMO / ENGINE_OFFER / ENGINE_TARGET) ready for sFTP.

Hard rule

Only widget defaults are edited. The body of NB06 is contract code — never regenerate it. If you think a widget is missing from the shared file, file an ADR; don't work around it locally.

docs

documentation

final step

Reads the audit trail + funnel + intake to render docs/README.md, docs/DOC.md, and the handovers HTML page. Always run last.

8Sentinel blocks — why your hand-edits survive

Every stage authoring pass wraps its work in a sentinel marker:

# region campaign-specific:data-prep — START (managed by campaign-orchestrator)
# ... custom CTE, WHERE conditions, derived columns ...
# endregion campaign-specific:data-prep — END

For SQL inside string literals, the equivalent is:

/* campaign-specific:data-prep:START */
  AND pb.anniv_year BETWEEN 5 AND 10
/* campaign-specific:data-prep:END */
What this guarantees

Re-running /campaign stage data-prep after you've hand-edited outside the sentinel block will only replace the managed region. Your hand-edits to the framework code, comments, or widgets outside the markers are preserved.

What this does NOT guarantee

If you hand-edit code inside the sentinel block, the next stage re-run will overwrite it. If you need to keep manual logic inside the managed region, move it out — or update the intake and let the orchestrator regenerate.

9Schema verification — the anti-hallucination layer

Before any SQL is authored, the relevant tables are passed through schema-verifier:

# under the hood, the orchestrator calls something like:
schema-verifier column ..agent_master.vndr_nm
# returns:
status: not_found
suggestion: vnd_nm

The verifier caches results to lt-memory/schemas/<fqn>.json with a 7-day TTL. Pass --refresh to force a re-fetch (e.g. after a known DDL change). Three known historical typo traps the verifier will catch:

WrongRightTable

paid_through_dtpt_thru_dtpolicies

10Deploy & verify loop

The deploy-verifier never uses databricks workspace import. The only acceptable deploy path is:

1. local change → git commit
git push origin main

2. trigger the Databricks Git folder pull
databricks repos update 985143326884916 --branch main

3. poll until workspace HEAD == local HEAD
databricks repos get 985143326884916 # .head_commit_id

4. submit one-shot job against the Analytics cluster
databricks jobs submit --json '{ ... }'

5. assert minimum row counts via SQL warehouse
SELECT COUNT(*) FROM <output_fqn>

11Troubleshooting

SymptomLikely causeFix
Orchestrator says "intake has TODOs remaining" and refuses to stage. The intake interview was skipped or partially answered. Re-run /campaign init <campaign> <batch> (it's idempotent past the bootstrap step) or hand-edit the remaining <!-- TODO --> markers in docs/intake.md.
Schema-verifier says warehouse_unavailable. SQL warehouse is stopped. Start DW_DBX_UC, or pass --cluster to fall back to a running cluster (slower & pricier).
Deploy-verifier reports sync_timeout. Workspace Git folder isn't pulling — usually a branch mismatch (workspace on a feature branch while you pushed to main). Switch the Databricks Git folder to main via the workspace UI, or push to the branch the workspace is tracking.
Job runs but row-count assertion fails. An exclusion is more aggressive than intended, or a join dropped rows. Don't blindly relax the assertion. Add temporary print(df.count()) probes between stages and run /campaign verify again.
assert_leads_list_schema() raises. A column was dropped or renamed in your sentinel block. The traceback names the missing/extra column. Either restore it, or update docs/extras_data_dictionary.csv if it's a genuinely new column.
Orchestrator wants to edit generic-campaigns/lib/. It found an exclusion in your campaign that appears in ≥2 campaigns and is recommending promotion. This requires your explicit confirmation via ask_user. Confirm only if you've checked the existing campaigns' SQL agrees.
You see an excl_shared_phone placeholder set to "N". You're looking at the most recent campaign batch or a clone of it. Known historical gap — batch-1 implements it in NB02 step 5.5. The orchestrator's intake-extractor flags this automatically.

12FAQ

Do I need to restart Copilot CLI after pulling new skill versions?

Yes — the CLI snapshots skills/ at session start. Mid-session changes aren't hot-reloaded.

Can I run a stage without going through /campaign?

Yes. The notebooks are normal Databricks notebooks; you can open them in the workspace UI and run them directly. The orchestrator just guarantees consistency. But if you skip verify, you also skip the audit-trail entry and the row-count assertion — your future self will thank you for not skipping.

What if my campaign needs something the template doesn't support?

Three escalation paths, in order of preference:

  1. Express it via the intake — most "special" requirements are just unusual values in section 2/3/4.
  2. Put it inside a sentinel block in the relevant notebook. The orchestrator preserves your hand-edits outside the block on re-runs.
  3. If two or more campaigns need the same thing, file an ADR under generic-campaigns/docs/adr/ proposing a framework change. Don't quietly modify lib/.

How does the orchestrator decide what's "campaign-specific"?

It reads generic-campaigns/template/intake.md as the canonical schema. Anything mentioned in those 8 sections is campaign-specific by definition. Anything not mentioned is framework-owned.

Why is the funnel artefact written twice (to Volume and docs/)?

Per ADR 0002: the Volume copy is canonical (regenerated every run); the docs/*funnel.{svg,png} snapshot is refreshed only at campaign launch via databricks fs cp. The docs/ copy is what goes into handovers and management decks.

Can I extend the orchestrator to non-campaign workflows?

v1 is intentionally narrow — campaigns under agency-campaigns/ only. A v1.1+ idea is to apply the same pattern to feature-mart/ pipelines and cx-projects/ ingestion notebooks. That's a future skill, not a campaign-orchestrator extension.
